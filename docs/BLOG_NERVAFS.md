# 当 AI Agent 遇见文件系统：NervaFS 的技术思考

> 基于 NervaFS 周会讨论的技术博客，探讨 AI Agent 基础设施的核心问题。

## 引言

2026 年的 AI 领域，Agent 已经不再是一个新鲜概念。从 Cursor 到 Claude Code，从 Manus 到各种 autonomous agent 框架，我们看到了一个清晰的趋势：**AI 正在从"回答问题"进化到"执行任务"。**

但在 Agent 能力飞速发展的同时，一个基础性问题始终没有解决：**Agent 之间的协作基础设施是什么？**

就像早期互联网需要 TCP/IP，早期云计算需要 Docker/K8s，Agent 时代也需要自己的基础设施层。NervaFS 的核心洞察是：**文件系统，可能是 Agent 基础设施的最佳隐喻。**

## 为什么是文件系统？

在讨论 NervaFS 之前，我们先看看现有 Agent 框架是如何处理核心问题的：

### 存储：Agent 的记忆放哪里？

工业界的主流方案是向量数据库——把 memory 和 knowledge 放在向量库中，通过 query 相似度检索记忆片段。但这带来了几个问题：

- **语义鸿沟**：向量相似 ≠ 语义相关，尤其在专业领域
- **调试困难**：你无法"cat"一个向量库，无法用 grep 搜索
- **版本管理缺失**：向量库天然不支持 diff、branch、revert

NervaFS 提出了另一种思路：**基于文件系统存储，用语义 grep 替代向量检索。** 具体来说：

- Agent 的 memory、knowledge、skill 都以文件形式组织
- 路径结构本身提供了天然的命名空间和权限控制
- 可以在文件系统之上构建向量索引，实现"语义 grep"
- 天然支持版本管理（Git）

### 通信：Agent 之间怎么说话？

现有的 Agent 通信方案百花齐放：

| 框架 | 通信方式 | 特点 |
|------|---------|------|
| Claude Code | 文件系统 + Git worktree | 用文件作为共享内存，Git 隔离任务 |
| AutoGen | 函数调用 | 紧耦合，同一进程内 |
| CrewAI | 消息队列 | 松耦合，但协议自定义 |
| LangGraph | 图节点传递 | 状态图模型，适合流程控制 |

问题在于：**每种实现方式都不一样，如何统一？**

NervaFS 团队认为，可以借鉴 Unix 哲学——**一切皆文件**。Agent 之间的通信可以通过文件系统操作来实现：

- **写信** = 写入文件到对方的 mailbox 目录
- **广播** = 写入共享 channel 目录
- **密送** = 写入对方私有目录，权限控制保证其他 Agent 看不到

这实际上回到了 Email 的 To/CC/BCC 模型。我们在 lingtai-node 的设计中发现，**Email 模型天然适合 Agent 通信**，因为：

1. **地址明确**：每个 Agent 有唯一地址，消息路由清晰
2. **CC/BCC 有意义**：多 Agent 协作时需要控制信息可见性
3. **异步友好**：Agent 不总是"在线"，邮件模型天然适合
4. **可审计**：inbox/sent/archive 提供完整通信记录

### 身份：Agent 是谁？

LingTai（灵台）系统给出了一个优雅的解决方案：**一个 Agent 的身份不是存储在某个中心数据库里的，而是分布在它的文件系统中。**

具体来说，一个 Agent 的"自我"由以下文件定义：

```
.agent/
├── identity/       # 我是谁——性格、专长、价值观
├── memory/         # 我记得什么——对话历史、经验教训
├── knowledge/      # 我知道什么——可验证的事实、关键决策
├── skill/          # 我会什么——可复用的流程和程序
├── communication/  # 我和谁说话——通信记录、协作关系
└── handover/       # 我如何交接——迁移和恢复的契约
```

**这些文件本身就构成了 Agent 的身份。** 迁移一个 Agent = 拷贝它的目录。备份一个 Agent = 打包它的文件系统。

## 渐进式披露：文件系统的精髓

NervaFS 周会中讨论了一个有趣的概念：**渐进式披露（Progressive Disclosure）**。

传统文件系统的访问是全有或全无的——你要么能读整个文件，要么不能。但对于 Agent 来说，很多场景需要更细粒度的控制：

- **Memory 访问**：Agent A 需要 Agent B 的部分记忆，但不应该看到全部
- **Skill 借鉴**：Agent A 想使用 Agent B 的某个技能，但不应该访问 B 的核心身份
- **协作上下文**：一组 Agent 共享工作上下文，但各自保留私密状态

NervaFS 的方案是：每个 Namespace 采用渐进式披露。访问 Memory、Skill 等路径时，根据需求渐进式访问——需要什么就给什么，不多不少。

这在一定程度上定义了 **Harness（工具链/框架）** 的边界：Harness 不应该是一个全知的中间人，而应该是一个渐进式披露的文件系统层。

## Memory vs Knowledge：拆还是合？

会上有一个精彩的讨论：

- **统一派**（邓鑫怀）：memory 和 knowledge 可以统一放在 state store 里，作为状态存储
- **分离派**（illyrix）：memory 是每个 agent 独立的，不需要共享；knowledge 可以共享——二者应拆开

我们的看法是：**从文件系统的角度，这个问题有天然答案。**

```
agent/
├── memory/          # 私有——600 权限，只有自己能访问
│   ├── conversations/
│   └── experiences/
└── knowledge/       # 共享——644 权限，他人可读
    ├── facts/
    └── decisions/
```

文件系统的权限模型天然区分了"私有"和"可共享"。不需要额外的中间件——**chmod 就是 API。**

## Agent 迁移性与 Hand-off：核心场景

经过讨论和投票，NervaFS 团队决定前期聚焦于两个核心场景：

1. **Agent 迁移性（Portability）**：Agent 能否在不同框架之间无缝迁移？
2. **Hand-off（交接）**：一个 Agent 能否将自己的状态和任务交接给另一个 Agent？

这两个场景直击 Agent 基础设施的核心痛点：

**迁移性**的核心挑战是协议碎片化。Claude Code 用文件系统，AutoGen 用函数调用，CrewAI 用消息队列——它们之间没有统一的语言。NervaFS 的方案是定义一个 **Node Contract**——一个跨框架的 Agent 身份描述规范，使得任何符合规范的 Agent 都可以在任何支持该规范的框架中运行。

**Hand-off** 的核心挑战是状态序列化。一个正在执行任务的 Agent 如何优雅地将自己的"心智状态"打包交给另一个 Agent？这需要：

- 明确哪些状态需要迁移（identity、memory、knowledge、skill）
- 明确哪些状态可以丢弃（临时对话、中间计算）
- 提供迁移过程中的版本兼容性保证

这也是 lingtai-node 正在做的事情——定义一套标准的 Agent Node Contract，让 Agent 可以像 Docker 容器一样"一次定义，到处运行"。

## 验证：如何知道 Agent 做对了？

会上讨论了两种验证模式：

**基于规则的验证**（如 Coding、SQL）：可以通过 PR 解决情况、准确率等客观指标来验证。这是 Agent 验证的"简单模式"——有 ground truth 可以对齐。

**基于判断的验证**（如专利生成、中医诊断）：没有客观正确答案，需要领域专家根据一定数量的 case 定制 rubrics（评分标准），然后验证 agent eval 和人类评分的一致性。

对于 NervaFS 来说，初期的验证应该聚焦于前者——**选一个可以客观验证的场景，做到极致。** 这也是会上达成的共识："Make the first thing boring sharp"——像 Linux 和 Git 一样，从解决核心问题出发，避免步子跨太大。

## 市场方向

开源项目的价值不在于直接变现，而在于提供基础设施级解决方案，吸引开发者和用户。

NervaFS 的市场策略应该是：

1. **效果驱动**：让 Agent 的迁移和交接真正 work，用 demo 说话
2. **低成本测试**：让用户可以极低成本地试用
3. **积累 edge case**：真实用户场景中的边缘情况是最好的优化素材
4. **社区驱动**：star 数和开源活跃度是 Agent 时代的新"声誉货币"

## 技术路线总结

NervaFS 的技术路线可以概括为三层：

```
┌─────────────────────────────────┐
│       应用层：Agent 协作场景       │
│   （迁移性、Hand-off、通信映射）    │
├─────────────────────────────────┤
│       协议层：Node Contract       │
│   （身份、记忆、知识、技能、交接）    │
├─────────────────────────────────┤
│       基础层：文件系统 + 语义搜索    │
│   （渐进式披露、权限控制、语义grep）  │
└─────────────────────────────────┘
```

先做好基础层——让文件系统成为 Agent 的原生存储和通信机制。然后在基础层之上构建协议层——定义跨框架的 Agent 身份规范。最后在协议层之上构建应用层——实现具体的迁移和交接场景。

## 结语

NervaFS 的核心洞察可以用一句话概括：**Agent 需要的不是更聪明的模型，而是更好的基础设施。**

文件系统是计算机科学中最成功的基础设施抽象之一。它简单、通用、可组合。如果把 Agent 的核心问题——存储、通信、身份、迁移——映射到文件系统的概念上，很多复杂的问题会变得清晰。

这不是说文件系统是万能的。向量检索在某些场景下仍然不可替代，消息队列在实时协作中仍然有优势。但作为一个基础设施层的核心隐喻，文件系统提供了一个足够简单、足够强大的起点。

就像 Linus Torvalds 当年用 "just a hobby, won't be big and professional" 来描述 Linux 一样——也许 Agent 的基础设施，就应该从这么一个简单的问题开始：**能不能让 Agent 像文件一样，可以被创建、移动、复制和组合？**
