# 通信模型映射：Email CC/BCC 与现代 IM 的对应关系

## 1. Email 的通信模型

### 1.1 基本字段

| 字段 | 语义 | 可见性 | 典型用途 |
|------|------|--------|----------|
| **To** | 主要收件人 | 所有人可见 | 需要执行/回复的人 |
| **CC** (Carbon Copy) | 抄送 | 所有人可见 | 需要知晓但不需要行动的人 |
| **BCC** (Blind Carbon Copy) | 密送 | 仅发件人和 BCC 自己可见 | 秘密抄送，其他人不知道 |

### 1.2 核心语义

Email 的通信模型基于**信件隐喻**：

```
发件人 → To:  这封信是写给你的，你需要读并可能回复
       → CC:  这封信也给你看，但你不一定要回复
       → BCC: 这封信偷偷给你看，别人不知道你也收到了
```

关键设计决策：
- **To ≠ CC 的行动暗示**：To 暗示"你需要做点什么"，CC 暗示"仅供参考"
- **BCC 的隐私保证**：To 和 CC 收件人看不到 BCC 列表
- **Reply vs Reply All**：Reply 只回复发件人，Reply All 回复所有人（To + CC）
- **异步通信**：不期望即时回复，适合长篇思考和正式沟通

## 2. Slack 的通信模型

### 2.1 通信单元

| 单元 | 类比 Email | 语义 |
|------|-----------|------|
| **Channel** | 邮件列表 (Mailing List) | 公开/私有的群组对话 |
| **DM** (Direct Message) | 私信 (To only) | 一对一私密对话 |
| **Thread** | Reply chain | 对某条消息的分支讨论 |
| **@mention** | To + CC | 引起特定人的注意 |

### 2.2 核心语义

Slack 的通信模型基于**空间隐喻**：

```
Channel（公开）→ 公司走廊：任何人都能看到和加入对话
Channel（私有）→ 会议室：只有被邀请的人能看到和加入
DM              → 隔间对话：只有两个人
Thread          → 角落里的分支讨论：不干扰主频道
@mention        → 拍肩膀：在嘈杂的房间里引起注意
```

关键设计决策：
- **频道是持续的**：不像邮件一封一封，频道是持续的对话流
- **@here vs @channel**：`@here` 只通知在线的人，`@channel` 通知所有人
- **Thread 隔离**：Thread 里的回复不会通知频道里的其他人
- **半同步通信**：期望比邮件更快的回复，但不一定是即时的

## 3. 飞书/Lark 的通信模型

### 3.1 通信单元

| 单元 | 类比 Slack | 类比 Email | 语义 |
|------|-----------|-----------|------|
| **群聊** | Channel | Mailing List | 群组对话 |
| **私聊** | DM | To (1:1) | 一对一对话 |
| **话题** | Thread | Thread | 分支讨论 |
| **@人** | @mention | To + CC | 引起注意 |
| **文档评论** | — | Reply with annotation | 在文档上下文中讨论 |
| **机器人消息** | Bot message | BCC (from system) | 系统推送的通知 |

### 3.2 核心语义

飞书的通信模型融合了**邮件 + IM**：

```
群聊 → 既是邮件列表（可发公告），又是聊天室（可实时对话）
私聊 → 既是私信，又是即时通讯
话题 → 既是邮件 Thread，又是论坛帖子
@人  → 既是 To/CC，又是"拍肩膀"
```

飞书的独特之处：
- **消息类型丰富**：文本、富文本、卡片（Interactive Card）、投票、文件
- **群公告** ≈ 发给所有人的邮件（所有人 @all）
- **话题回复** ≈ Reply All，但可以折叠
- **机器人** ≈ 自动化的 BCC 发送者
- **已读/未读状态**：介于邮件和 IM 之间

## 4. 映射关系

### 4.1 Email → Slack/飞书 映射表

| Email 概念 | Slack 对应 | 飞书对应 | 差异说明 |
|-----------|-----------|---------|----------|
| **To** | `@mention` 在 channel 或 DM | `@人` 在群聊或私聊 | Email To 是强制的，IM @是可选的 |
| **CC** | Channel 消息（不 @） | 群聊消息（不 @） | Channel 里所有人都能看到 = CC |
| **BCC** | 无法直接实现* | 无法直接实现* | IM 天然没有 BCC 概念 |
| **Reply** | DM 回复 / Thread 回复 | 私聊回复 / 话题回复 | Thread 是 Slack 发明的概念 |
| **Reply All** | 在 Channel 回复 | 在群聊回复 | 等价于在原始频道继续讨论 |
| **Subject** | Channel 名称 / Thread 标题 | 群聊名称 / 话题标题 | IM 没有显式 Subject 字段 |
| **Forward** | Share message to another channel | 转发到另一个群 | 等价 |
| **Attachment** | File upload | 文件上传 | 等价 |
| **Priority** | `@channel` / `@here` | `@all` / 紧急消息 | IM 用 @级别代替 Priority header |

### 4.2 BCC 在 IM 中的替代方案

BCC 是 Email 独有的概念，IM 中没有直接等价物。替代方案：

| BCC 用途 | Slack 替代 | 飞书替代 |
|---------|-----------|---------|
| **秘密通知某人** | DM 截图/转发 | 私聊转发 |
| **邮件合并（群发单显）** | Bot DM 每个人 | 机器人私聊每个人 |
| **列表保密** | Private channel | 私有群聊 |
| **审计/记录** | Admin export | 管理后台导出 |

### 4.3 通信模式映射

| 通信模式 | Email 做法 | Slack 做法 | 飞书做法 |
|---------|-----------|----------|---------|
| **1:1 正式沟通** | To: 对方 | DM | 私聊 |
| **团队通知** | To: 全员, CC: 相关人 | `@channel` in team channel | `@all` in team group |
| **跨团队协作** | To: 负责人, CC: 各团队代表 | Cross-post to multiple channels | 转发到多个群 |
| **秘密沟通** | BCC | DM / Private channel | 私聊 / 私有群 |
| **讨论/决策** | Reply All thread | Thread in channel | 话题 in group |
| **异步长文** | 邮件正文 | Canvas / Doc | 飞书文档 + 评论 |
| **实时讨论** | 不适合 | Huddle / Channel chat | 音视频 / 群聊 |

## 5. 深层设计哲学对比

### 5.1 推 vs 拉

| 模型 | 推模式 (Push) | 拉模式 (Pull) |
|------|-------------|-------------|
| **Email** | 发件人推到收件箱 | 收件人主动读邮件 |
| **Slack** | @mention 推通知 | 用户浏览 channel |
| **飞书** | @人 推通知 / 机器人推 | 用户浏览群聊 |

Email 是**强推**——所有收件人都会在收件箱里看到。
IM 是**弱推**——只有在 @你 的时候才推送，否则需要主动去看。

### 5.2 持久性

| 模型 | 消息持久性 | 搜索性 | 上下文保留 |
|------|----------|--------|----------|
| **Email** | 永久 | 强（每个客户端都支持） | 强（Thread 视图） |
| **Slack** | 永久（付费）/ 90天（免费） | 强 | 中（Thread 可能脱节） |
| **飞书** | 永久 | 强 | 中（话题可以脱节） |

### 5.3 正式程度

```
Email ─────────── Slack/飞书 ─────────── 即时通讯(微信/WhatsApp)
  ↑ 正式                                    ↑ 随意
  - 格式化                                   - 自由格式
  - 期待结构化回复                            - 期待快速回复
  - 有 Subject/Signature                    - 没有固定结构
  - 异步为主                                  - 同步为主
```

## 6. Agent 通信中的应用

在 Agent 网络中，我们使用 Email 模型（LingTai 的 `email` intrinsic），因为它：

1. **地址明确**：每个 Agent 有唯一地址，消息路由清晰
2. **CC/BCC 有意义**：多 Agent 协作时需要控制信息可见性
3. **异步友好**：Agent 不总是"在线"，邮件模型天然适合
4. **可审计**：inbox/sent/archive 提供完整的通信记录

映射到 IM 的挑战：
- Agent "群聊" = broadcast channel（如 Telegram group、飞书群）
- Agent "私聊" = 1:1 email
- Agent "@人" = To 字段
- Agent "BCC" = 目前 IM 无法直接支持

## 7. 总结

| 维度 | Email | Slack | 飞书 |
|------|-------|-------|------|
| **核心隐喻** | 信件 | 房间 | 信件 + 房间 |
| **寻址** | To/CC/BCC | @mention / channel | @人 / 群聊 |
| **隐私** | BCC | Private channel | 私有群 |
| **同步性** | 异步 | 半同步 | 半同步 |
| **持久性** | 强 | 中~强 | 强 |
| **正式度** | 高 | 中 | 中 |
| **适合 Agent** | ✅ 天然适合 | 需要适配 | 需要适配 |

**一句话总结：** Email 的 To/CC/BCC 是基于信件的精确寻址模型，IM 的 @/channel 是基于空间的粗粒度寻址模型。前者精确但繁琐，后者便捷但缺乏细粒度控制。Agent 通信需要精确控制，所以 Email 模型更适合。
